## One-line angle

Agents are policy-governed decision runtimes — not autonomous loops with API keys.

## Opening thesis

Enterprise agent hype focuses on autonomy. G.A.I.N Agents focuses on **governed orchestration**: explicit planning boundaries, tool registries, policy gates, and audit trails — so agents scale in regulated environments.

---

## Why this domain needs G.A.I.N

Agents multiply risk faster than chat:

- Every tool call is a potential compliance event
- Every plan step needs traceability
- Every escalation path must be designed — not improvised

An Insight argues *where guardrails execute*. This article maps the **full agent architecture** through G.A.I.N.

---

## Grounded (G)

**Patterns**

- Tool allowlists per role, tenant, and use case — not open-ended function calling
- Context assembly from authorized sources only (RAG, APIs, memory — all scoped)
- Ground truth for "what happened" — immutable action log

**Anti-patterns**

- Agents with direct database write access
- Tool definitions scattered across prompts

---

## Adaptive (A)

- Eval harnesses for plan quality, tool selection accuracy, escalation rate
- Shadow mode: agent plans but human approves actions in early rollout
- Production overrides feed policy updates — not silent prompt patches

---

## Intelligent (I) — dominant

**Runtime flow**

```text
Request → Planner → Policy gate → Tool registry → Execute → Audit → Response
                ↓                      ↓
           Human review          Deny / escalate
```

**Patterns**

- **Planning vs execution separation** — plan is inspectable before tools run
- **Policy gates** at every tool boundary — business rules, not LLM judgment
- **Decision model integration** — when to automate vs escalate (see [Decision Model blueprint](https://jitendersharma.dev/blueprints/decision-model))
- **Workflows vs autonomy** — most "agents" in production are governed workflows

**Decisions**

- Human-in-the-loop is an architecture slot — not a UI checkbox
- Memory tiers: session · episodic · long-term — each with retention policy

---

## Native (N)

- Horizontal scale of stateless planners · durable workflow engine for execution
- Observability: plan steps, tool args, policy outcomes, latency per step
- Cost caps and circuit breakers on tool-heavy paths
