# G.A.I.N Template & Example

Use when **planning or drafting** a Signature Article on [/frameworks/gain-*](https://jitendersharma.dev/frameworks). Catalog → **[Types](./Types#gain-signature-articles)**.

:::tip[Not an Insight]
Signature Articles — **no** Core pillar tag · **no** primary voice tag · **no** Content Bank Insights entry.
:::

---

## Planning block

Copy when adding to the [Types catalog](./Types#gain-signature-articles):

```text
Article: G.A.I.N [Topic]
Route: /frameworks/gain-[topic]
G.A.I.N pillar: [G / A / I / N — one dominant letter]
One-line angle: [how this topic works through G.A.I.N in one sentence]
Opening thesis: [architectural claim in one sentence — not hype]
Sections: Grounded (G) · Adaptive (A) · Intelligent (I) · Native (N)
May draw from: [Insight titles — optional]
Status: 📋 Planned
```

---

## Page structure

Every published Signature Article on site follows this outline (match [frameworks overview](https://jitendersharma.dev/frameworks) style):

```text
1. Link back to core G.A.I.N framework
2. One-line angle + opening thesis (architectural, not hype)
3. Why this domain needs G.A.I.N (not generic AI advice)
4. Grounded (G) — patterns · decisions · anti-patterns
5. Adaptive (A) — feedback · eval · rollout
6. Intelligent (I) — reasoning · orchestration · boundaries
7. Native (N) — scale · ops · observability · compliance
8. Related Insights (optional) · Blueprints · core framework
```

## Naming and voice

1. **Name it consistently** — G.A.I.N (with periods); spell out "Governed AI-Native Systems" on first use
2. **Show the flow** — mermaid diagrams and layer tables (match framework page style)
3. **One Signature Article at a time** — finish G.A.I.N LLM before starting the next domain
4. **Keep channels separate** — Insights → [Tagging Guidelines](../00 content/50 Tagging-Guidelines) · Signature Articles → [Types](./Types)
5. **Link back** — Signature Articles ↔ related Insights ↔ core framework
6. **Architectural, not marketing** — problem/tension titles, not hype
7. **Systems over models** — redirect from model debates to architecture decisions
