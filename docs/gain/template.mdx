# G.A.I.N Template

Use when **planning or drafting** a Signature Article on [/frameworks/gain-*](https://jitendersharma.dev/frameworks). Catalog → [Types](./Types) · channel rules → [Gain Engine](./Overview) · worked examples → [Example](./Example).

:::info[Not an Insight]
No Core pillar tag · no primary voice tag · no Content Bank Insights entry. Tag one dominant G.A.I.N letter.
:::

---

## Planning block

```text
Article: G.A.I.N [Domain]
Route: /frameworks/gain-[domain]
Dominant pillar: [G / A / I / N — one, occasionally two]
One-line angle: [how this domain works through G.A.I.N in one sentence]
Opening thesis: [architectural reframe — not hype]
Sections: Grounded (G) · Adaptive (A) · Intelligent (I) · Native (N)
May draw from: [Insight titles — optional]
Blueprint sibling: [/blueprints/... — optional]
Playbook (later): [/playbooks/... — optional]
Status: 💡 Idea | 📋 Planned | 📝 Draft | ✅ Live
```

## Page structure (published MDX)

### Frontmatter

```yaml
---
slug: gain-[domain]
title: G.A.I.N [Domain]
description: [One-line angle — architectural, not marketing.]
# No Insight tags — Signature Articles are untagged
authors: [iamsharmajitender]
---
```

### Body outline

```text
1. Link back to core G.A.I.N framework (/frameworks)
   Spell out "Governed AI-Native Systems" on first use.

2. One-line angle + opening thesis (architectural, not hype)
   Reframe systems over models — quotable, not vendor pitch.

3. Why this domain needs G.A.I.N (not generic AI advice)
   Name the enterprise failure mode this domain fixes.

4. Domain on one page
   One layer table or flow diagram for the whole domain before G·A·I·N sections.
   Flag dominant pillar (G / A / I / N) — lead with or expand that section.

5–8. Grounded (G) · Adaptive (A) · Intelligent (I) · Native (N)
   Each pillar section (see below). Add a diagram on the dominant pillar.

9. Related assets — Insights · Blueprint · Playbook · core framework
```

### Each pillar section

Use the same shape for G, A, I, and N:

```text
## [Pillar name] ([letter])

**Patterns** — what good looks like in this domain
**Decisions** — what architects choose (citeable)
**Anti-patterns** — what breaks at enterprise scale

[Optional flow diagram or layer table — required on the dominant pillar]
```

## Voice & tone

| Do | Don't |
| --- | --- |
| Reframe to "which model, which task, which policy, what cost" | Compare model brands |
| Map the **whole domain** layer by layer | Restate a single-tension Insight |
| Write as **your** G.A.I.N framework | Write a generic "intro to RAG/agents" |
| Spell out "Governed AI-Native Systems" on first use | Leave G.A.I.N undefined |
| Lead with the **dominant pillar** | Give all four pillars equal weight by default |

**No em dashes (`—`)** in prose, tables, or mermaid.

---

## Before publishing

- [ ] **One dominant G.A.I.N pillar** — G · A · I · N (lead or expand that section)
- [ ] **Untagged** — no Core pillar or voice tags, no Insights entry
- [ ] **Frontmatter** — slug · title · description (no Insight tags)
- [ ] **One-line angle + opening thesis** present (architectural, not hype)
- [ ] **Why this domain needs G.A.I.N** section present
- [ ] **Domain on one page** — one article-level layer table or flow diagram
- [ ] **All four pillars** mapped: Patterns · Decisions · Anti-patterns each
- [ ] **Diagram on dominant pillar** (article-level diagram minimum)
- [ ] **No em dashes** in prose, tables, or mermaid
- [ ] Cross-links: core `/frameworks` + related Insights + Blueprint sibling
- [ ] Updated in [Types catalog](./Types)
