# ARC — Architecture

**Core question:** What is the architecture pattern or flow?

Use this voice when you want to explain how a system works — the flow, the layers, and the design principles that make it hold up at scale.

:::info[Insight + LinkedIn]
Each Insight uses **one primary voice**. The paired **LinkedIn voice post** uses the **same voice** and **same tags** — one post per Insight, shorter format, link back to site.
:::

## When to use ARC

- You are documenting or teaching a system design
- The reader needs to understand how components connect
- You want to establish technical authority through structure, not opinion alone

## Structure

Follow this flow in order:

1. **Problem statement** — What needs solving
2. **Architecture flow** — Simple, ordered breakdown of how the system works
3. **Key layers** — What each layer is responsible for
4. **Design principles** — The truths that govern good design

## Template

Copy this block and replace the bracketed placeholders:

```text
Here's how I think about [system or problem]:

Flow:
[Step 1] → [Step 2] → [Step 3] → [Step 4] → [Step 5]

Why this matters:

* [Layer 1] controls [responsibility]
* [Layer 2] controls [responsibility]
* [Layer 3] controls [responsibility]

Key principle:
[Architecture truth]
```

## Examples

### Example 1 — RAG

> Here's how I think about enterprise RAG systems:
>
> **Flow:**
> User → Identity → Policy → Retrieval → Validation → LLM → Response
>
> **Why this matters:**
>
> - Identity scopes retrieval
> - Policy scopes access
> - Retrieval scopes relevance
> - Validation scopes correctness
>
> Without this, RAG becomes a probabilistic search engine.
>
> **Key principle:** Similarity should never own truth.

### Example 2 — Agents (G.A.I.N)

> Here's my reference model for governed agentic systems (G.A.I.N):
>
> **Flow:**
> User → Identity → Policy → Context Builder → Agent → Tools → Audit Layer
>
> **Layer responsibilities:**
>
> - Governance defines constraints
> - Agents define plans
> - Tools execute deterministically
> - Audit tracks everything
>
> **Key principle:** Agents orchestrate. Platforms govern.

## Pre-publish checklist

- [ ] States the problem before presenting the architecture
- [ ] Uses a clear, linear flow the reader can follow
- [ ] Assigns a responsibility to each layer
- [ ] Includes at least one design principle
- [ ] Avoids marketing language — let the structure speak
